# CO2 Footprint Calculator

Static calculator for operational emissions from invoices and internal activity, aligned to GHG Protocol Scope 1, Scope 2, and Scope 3.

## What changed in this refactor

The app now uses a more structured data model so calculations are safer and easier to extend:

- Entries store a `factorRef` instead of relying only on `module + activity + scope`
- Factors store `category`, `subcategory`, `method`, `inputUnit`, `source`, `region`, and `validFrom`
- Buildings are stored separately with `grossFloorAreaM2`, ownership, energy control, and heating type
- Building records can link to a building so the app calculates **kg CO2e / m²**
- Unit mismatch validation blocks incorrect calculations from entering totals
- Legacy CSV imports still work and are normalized on import

## Scope coverage

- Scope 1: direct building fuel use such as on-site gas or stationary fuel
- Scope 2: purchased electricity and purchased heat
- Scope 3: travel, waste, and personnel activity

## Modules

- Building: can contain Scope 1 and Scope 2 records
- Personnel: Scope 3 only, including commuting and home office
- Waste: Scope 3 only
- Travel: Scope 3 only, separate from commuting

## Data model

### Emission record

Core fields used by the app:

- `module`
- `category`
- `subcategory`
- `ghgScope`
- `factorRef`
- `quantity`
- `unit`
- `buildingId`

### Emission factor

Stored with:

- `key`
- `module`
- `category`
- `subcategory`
- `ghgScope`
- `method` (`activity` or `spend`)
- `inputUnit`
- `value`
- `source`
- `region`
- `validFrom`

### Building

Stored with:

- `id`
- `name`
- `grossFloorAreaM2`
- `ownershipType`
- `energyControl`
- `heatingType`

## How to use

1. Open `index.html` in a browser.
2. Import `sample-invoices.csv` or your own CSV.
3. Add or edit buildings so building records can calculate intensity correctly.
4. Adjust emission factors on the right.
5. Add manual records for anything not present in invoices.

## CSV format

Accepted legacy columns:

`date,vendor,description,module,ghg_scope,activity,amount,quantity,unit,building_id,notes`

Preferred future format:

`date,vendor,description,module,category,subcategory,ghg_scope,activity,factor_ref,calculation_method,amount,quantity,unit,building_id,notes`

## Notes

- Travel and employee commuting should stay separate.
- Do not count lease rent as emissions if underlying utilities are already tracked.
- Changing factors recalculates records live. If you need locked reporting later, store an applied-factor snapshot per record.


## Latest refactor

This version adds:
- owner-controlled vs tenant-attributed building splits using `owner_share_pct` and `tenant_share_pct`
- commuting vs business travel metadata using `transport_mode`, `traveler_count`, `round_trip`, `employees_covered`, and `survey_period`
- building-level leased area support with owner/tenant summaries in the dashboard
